# XBotv2 Stage 2 Plan (Historical)

> Archived implementation plan. This is not a current specification; see
> `docsv2/README.md`.

## Confirmed Decisions

- Remove the multi-personality runtime design.
- Do not preserve backward compatibility with personality-based configs or session-internal workspaces.
- Use a general system prompt model.
- Read both global system config and workspace `AGENTS.md`.
- Treat the process start directory as the workspace, like OpenCode.
- Default to a new session on each start.
- Resume only when a session id is explicitly provided.
- If an explicit session id does not exist, fail with an error.
- Allow one server to host sessions from multiple workspace roots.
- `/provider`, `/permission`, and `/sandbox` default to session scope.
- External workspace access defaults to `read=ask`, `write=deny`.
- The agent must be able to request external access actively, and the TUI must support approving or denying that request.

## Runtime Model

### System Prompt

System prompt sources, in order:

1. `data/config/system.yaml`
2. Workspace `AGENTS.md`
3. Runtime safety/context sections generated by XBotv2

The effective prompt is the concatenation of these sources. Missing `AGENTS.md` is allowed.

### Config Files

Target config layout:

```text
data/config/system.yaml
data/config/providers.yaml
data/config/permissions.yaml
data/config/sandbox.yaml
```

No personality config path remains in the runtime.

### Session State

Default startup creates a new session id.

Explicit session resume requires the session to exist.

Session state records:

```yaml
session_id: ...
thread_id: agent
workspace_root: /path/where/xbotv2/started
provider: default
permission_overrides: {}
sandbox_overrides: {}
```

## Workspace Model

- `workspace_root` is the current working directory unless `--workspace` is provided.
- Filesystem and shell tools operate relative to `workspace_root`.
- No `sessions/<sid>/workspace` directory is created.
- Session persistence stays under `data/sessions/<sid>/state`.
- Sandbox policy, not a fake workspace directory, controls access boundaries.

## Permission Model

Effective permission policy merges:

```text
built-in defaults
-> global config
-> session overrides
-> one-shot live decisions
```

Default scope for command changes is `session`.

Supported command examples:

```text
/permission status
/permission list
/permission set shell allow
/permission set shell allow --scope session
/permission reset shell
/permission reset --scope session
```

## Sandbox Model

Defaults:

```yaml
external_read: ask
external_write: deny
workspace_read: allow
workspace_write: allow
```

The agent can request external access by attempting a sandboxed tool call outside the workspace. The server emits `permission_request` with `source=sandbox`; the TUI renders an approval choice. Session approval writes a session-level sandbox override.

Supported command examples:

```text
/sandbox status
/sandbox enable
/sandbox disable
/sandbox set external_read ask
/sandbox set external_write deny
/sandbox reset --scope session
```

## Provider Model

Provider config is independent of personality.

Target layout:

```yaml
default: deepseek
providers:
  deepseek:
    provider: openai
    model: deepseek-chat
    base_url: https://api.deepseek.com/v1
    api_key_env: DEEPSEEK_API_KEY
```

Default `/provider use ...` scope is session.

Supported command examples:

```text
/provider list
/provider status
/provider use deepseek
```

Provider switching during an active turn must fail or be deferred; first implementation should reject while a turn is active.

## Server Commands

The server owns command metadata and execution for runtime controls.

Endpoints:

```http
GET /commands
GET /sessions/{sid}/commands
POST /sessions/{sid}/commands
```

Minimum commands:

```text
/status
/provider
/permission
/sandbox
```

Command metadata includes name, slash form, description, argument hints, and examples. TUI uses this for completion and `/help`.

Local TUI commands remain:

```text
/exit
/clear
/help
```

Server command results do not enter LLM message history. They render as transcript notices and append protocol events.

## HTTP Session API Changes

`POST /sessions` request:

```json
{
  "session_id": null,
  "thread_id": "agent",
  "workspace_root": "/repo/project",
  "mode": "new"
}
```

Supported modes:

- `new`: create a new generated session id; fail if explicit id already exists.
- `resume`: require an existing session id; fail if missing.

Default mode is `new`.

Response:

```json
{
  "session_id": "20260606-143012-a8f2",
  "thread_id": "agent",
  "status": "ready",
  "workspace_root": "/repo/project",
  "agent_name": "XBotv2"
}
```

## TUI Changes

Startup sequence:

1. `GET /health`
2. `POST /hello`
3. `GET /commands`
4. `POST /sessions` with `workspace_root` and mode
5. Initialize local + server command completion/help

Slash dispatch:

- local commands run locally
- server commands call `POST /sessions/{sid}/commands`
- unknown commands render a notice

## Implementation Order

1. Replace personality config loading with system config loading.
2. Add workspace-root session creation and remove internal workspace creation.
3. Update CLI, TerminalSession, HTTP transport, dispatcher, and bootstrap session parameters.
4. Add session-level provider/permission/sandbox state helpers.
5. Add sandbox external access request behavior and TUI approval flow.
6. Add server command registry and command endpoints.
7. Update TUI command discovery, completion, help, and slash dispatch.
8. Replace personality/workspace tests with Stage 2 tests.

## Test Coverage Targets

- Default `POST /sessions` returns a generated new session id.
- Explicit missing session id with resume mode returns 404.
- One server can open sessions with different `workspace_root` values.
- `AGENTS.md` content appears in context.
- Filesystem relative paths resolve under startup workspace.
- Shell runs with cwd set to workspace root.
- External read emits sandbox permission request.
- External write denies by default.
- Session sandbox approval enables the requested external path only for that session.
- `GET /commands` drives TUI completion and `/help`.
- `/provider`, `/permission`, `/sandbox`, `/status` execute as server commands and do not enter LLM history.
